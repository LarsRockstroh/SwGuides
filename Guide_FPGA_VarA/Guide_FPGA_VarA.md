## Development in VHDL-2008

### Rules
* In FSMs, use "when others" for the reset state.
* Keep all signal assignments inside a process.
* In design files, do not initialize signals.
* Put the reset if-clause to the end of the process.
* Only one assignment per line.
* Call `finish` at the end of simulation.

### Naming Conventions

| Category                  | Convention            | Example                         |
|---------------------------|-----------------------|---------------------------------|
| Input ports               | `i_` prefix           | `i_Clk`, `i_Reset`, `i_En`      |
| Output ports              | `o_` prefix           | `o_Count`, `o_Initial_Ready`    |
| Internal signals          | `s_` prefix           | `s_Count_Reg`, `s_State`        |
| Generics                  | `G_` prefix           | `G_BitWidth`                    |
| Constants                 | `C_` prefix           | `C_BitWidth`                    |
| Types                     | `t_` prefix           | `t_State`                       |
| Enumeration values        | `e_` prefix           | `e_Idle`, `e_Counting`          |
| Variables   	            | `v_` prefix           | `v_Threshhold`                  |
| Procedures  	            | `p_` prefix           | `p_Sort`                        |
| Processes                 | `_P` postfix          | `Main_P`, `Clk_P`, `Stim_P`     |
| Functions  	            | `f_` prefix           | `f_Accumulate`                  |
| Entities  	            | Capital first letter  | `Median9`                       |
| Testbench entities        | `tb_` prefix          | `tb_Median9`                    |
| Filenames                 | Match entity name     | `Median9.vhd`, `tb_Median9.vhd` |
| Minimum identifier length | 4 characters          | `v_Arr` not `v`                 |

## Coding Recommendations

### Processes in Design Modules (for Synthesis)
Only use the following two notations of processes in design code.
#### Combinational Process (no signal inside the process is clocked)
```
CombinationalOne_P : process(all)
  -- variable definitions
begin
  -- Signal assignments and
  -- variable assignments
end process CombinationalOne_P;
```

#### Clocked Processes (all signals inside the process are clocked)
```
SequentialOne_P : process(i_Clk)
  -- variable definitions
begin
  if rising_edge(i_Clk) then
    -- Signal assignments and
    -- variable assignments

    if (i_Reset = '1') then
      -- Signal assignments during reset
    end if;
  end if;
end process SequentialOne_P;
```

### FSM Implementation
```
architecture Behavioral of FSM_Example is
  type t_State is (e_State1,
                   e_State2,
                   e_State3,
                   e_State_Reset);
  signal s_State : t_State;

begin
  FsmBasic_P : process(i_Clk)
  begin
    if rising_edge(i_Clk) then
        case s_State is
          when e_State1 =>
            s_State <= e_State2;
          when e_State2 =>
            s_State <= e_State3;
          when e_State3 =>
            s_State <= e_State1;
          when others =>   -- Includes reset state
            s_State <= e_State_Reset; 
        end case;          
          
      if (i_Reset = '1') then
        s_State <= e_State_Reset;
      end if;
    end if;
  end process FsmBasic_P;
end Behavioral;
```

### Interfaces
#### Enable Handshake/Interface
* Interface to transmit data from a transmitting VHDL module to a receiving VHDL module
* Ports

```
Transmitter Ports                      Receiver Ports
o_Xyz_Enable : out std_logic           i_Xyz_Enable : in std_logic
o_Xyz_Data   : out std_logic_vector    i_Xyz_Data   : in std_logic_vector
```

* Conventions
    * "Data" is considered as transmitted for every clock cycle when "Enable" = '1'
    * Receiver has no mechanism to pause or slow down the data transfer
    * If the receiver cannot process the "Data" word when "Enable" = '1' then this "Data" word is lost
    * Sometimes, "Enable" is named/labeled as "Valid"
        
#### Valid-Ready Handshake
* Interface to transmit data from a transmitting VHDL module to a receiving VHDL module
* Ports

```
Transmitter Ports                      Receiver Ports
o_Xyz_Valid  : out std_logic           i_Xyz_Valid  : in  std_logic
i_Xyz_Ready  : in  std_logic           o_Xyz_Ready  : out std_logic
o_Xyz_Data   : out std_logic_vector    i_Xyz_Data   : in  std_logic_vector

```

* Conventions
    * "Data" is considered as transmitted when "Valid" and "Ready" are both '1' in the same clock cycle
        * "Data" is not considered as transmitted in case of "Valid" = '0' or "Ready" = '0'
    * Typical: Once the transmitter has set "Valid" to '1' (assert),
    it is not allowed to set "Valid" to '0' (deassert) and it is not allowed to change "Data" until "Ready" was '1'
* Valid-Ready handshake is used in AMBA 4 AXI4-Stream Protocol
    * See section "2.2 Transfer signaling"
    * Data sheet: "IHI0051A_amba4_axi4_stream_v1_0_protocol_spec.pdf"
    * Source: [https://developer.arm.com/documentation/ihi0051/a/](https://developer.arm.com/documentation/ihi0051/a/)

* Header comments
* Used libraries and used packages
* Entity: Name of Entity equals filename

* Architecture

        * Definition of constants

### Type Casting and Type Conversion
#### Type Casing
* Changes interpretation of a bit vector

```
Object          Function                        Object

|signed|    --> std_logic_vector(signed)    --> |std_logic_vector|
|signed|    <-- signed(std_logic_vector)    <-- |std_logic_vector|
										    								    
|unsigned|  --> std_logic_vector(unsigned)  --> |std_logic_vector|
|unsigned|  <-- unsigned(std_logic_vector)  <-- |std_logic_vector|
```

#### Type Conversion (to_*)
* Changes representation inside development tool between integer and bit vector
* Integer is implemented as 32 bit or 64 bit

```
Object          Function                         Object

|signed|    --> to_integer(signed)          --> |integer|
|signed|    <-- to_signed(integer, Length)  <-- |integer|

|unsigned|  --> to_integer(unsigned)        --> |integer|
|unsigned|  <-- unsigned(integer, Length)   <-- |integer|
```


### Concurrent Signal Assignment
* Within architecture
    * Signal assignments to different signals inside architecture take place at the same time
    * Assigning to the same signal multiple times is not allowed in design
    * The following also counts as signal assignment (driving a signal) inside an architecture:
        * Mapping signal to an out port during instantiation
        * Driving the signal inside a process inside that architecture
            * Thus, driving a signal in more than one process is not allowed in design

* Within process
    * Signal assignments to different signals inside process also take place at the same time
    * But multiple signal assignments to the same signal are allowed
        * The last signal assignment for each signal is effective

### Attributes (selection)
|Attribute        |  Meaning                                                   |
|-----------------|------------------------------------------------------------|
|s_Signal'length  |  Number of bits in vector or number of elements in array   |
|s_Signal'high    |  Index of highest bit in vector                            |
|s_Signal'low     |  Index of lowest bit in vector (typically 0)               |

### Component Declarations (Don't)

    * Internal modules can be instantiated without component declaration

### Integrated Logic Analyzer (ILA) - Digital Oscilloscope
* Probes signals inside the FPGA and provides waveforms like simulation
* Approach
    * Allocate attribute вҖңmark_debugвҖқ to signals in architecture before begin
        * `attribute mark_debug : string;`
        * `attribute mark_debug of <signal>: signal is "true";`
    * Synthesize
    * Setup debug probe and save (information gets saved into xdc file)
    * Generate Bitstream
    * Additional .ltx file is generated with probe information to be used in Hardware Manager
